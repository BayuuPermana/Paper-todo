# Research: CSS Hover Reveal Patterns for Subtasks

**Date**: 2026-01-18

## 1. Executive Summary
The subtask UI is currently cluttered with three action buttons (Edit, Focus, Delete) visible at all times. Implementing a CSS-driven hover reveal will clean up the "Paper" aesthetic while maintaining functionality. The "Focus" button (ðŸŽ¯) requires special handling to remain visible when active.

## 2. Technical Context
- **Component**: `src/SubTaskItem.jsx`.
- **CSS Class**: `.sub-task-item` (parent), `.todo-actions` (child container).
- **Inline Styles**: Currently using inline styles for layout and individual button opacity (Line 60, 72, 83).

## 3. Findings & Analysis
- **Hover Logic**: CSS pseudo-classes (`:hover`) on the parent `.sub-task-item` can target child buttons.
- **Opacity States**:
    - Default: `0` (hidden).
    - Parent Hover: `0.6` (available).
    - Button Hover: `1` (focused).
- **Persistence Logic**: The `isFocused` prop in React determines if the ðŸŽ¯ button should bypass the hidden state.
- **Transitions**: `transition: opacity 0.2s ease-in-out` will prevent jarring "pop-in" effects.

## 4. Technical Constraints
- **Touch Support**: Hover doesn't exist on mobile. We must ensure buttons are visible or easily revealable on touch devices (Standard behavior: they will likely show on tap or we can use a media query to keep them visible on small screens).
- **Specificity**: CSS classes will override or augment existing inline styles.

## 5. Architecture Documentation
- Pattern: Parent-container-driven hover reveal.
- Required: Move some inline styles to `index.css` for cleaner control.
